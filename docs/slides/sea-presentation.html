Schema: The Foundation

Schema ‚Üí Collections ‚Üí Query Planning ‚Üí Agentic Search

// tools.schema.ts { "collections": { "tools": { "fields": ["name", "description"], "purpose": "semantic identity" }, "functionality": { "fields": ["features", "capabilities"], "purpose": "what it does" }, "usecases": { "fields": ["scenarios", "workflows"], "purpose": "when to use it" }, "interface": { "fields": ["deployment", "integration"], "purpose": "how to use it" } }, "vocabularies": { "categories": [...], "pricingModels": ["Free", "Paid"] } }

Why Schema-Driven?

  • Defines collections: Schema decides what deserves separate embeddings
  • Enables routing: Query planner knows where to search
  • Validates LLMs: Controlled vocabularies catch hallucinations
  • Portable: New domain = new schema file

Dynamic Result Fusion

The query planner decides how to merge results

LLM Chooses the Strategy

  • 3+ sources ‚Üí RRF (rank-based)
  • 2 sources ‚Üí Weighted Sum
  • 1 source ‚Üí No fusion needed

Fusion method is part of the query plan, not hardcoded

Why Dynamic?

  • Simple queries don't need complex fusion
  • Multi-collection queries benefit from RRF
  • Hybrid (vector + structured) needs special handling
// query-executor.node.ts finalCandidates = fuseResults( candidatesBySource, executionPlan.fusion // LLM decided: 'rrf' | 'weighted_sum' | 'none' );

The agentic part: even fusion strategy is planned, not prescribed

Plan Caching

The Problem

  • LLM calls are expensive
  • Identical queries waste compute
  • Latency adds up

The Solution

  • Hash the query (MD5)
  • Store intent + execution plan
  • Exact match lookup

What Gets Cached

  • Original query + hash
  • Extracted intent state
  • Query execution plan
  • Result candidates

Why Not Semantic Matching?

  • "free code editors" ‚âà "paid code editors"
  • Similar embeddings, opposite filters!
  • Exact match prevents wrong results

Cache hit = skip 2 LLM calls (intent + planner) ‚Üí significant cost savings

What I'm Still Figuring Out

Honest questions from building this...

Evaluation is Hard

  • How do I know if results are "good"?
  • User clicks? But users click bad results too
  • No standard benchmarks for agentic search

Schema Evolution

  • New AI tool categories appear monthly
  • When do I add vs. merge categories?
  • Re-embedding everything is expensive

LLM Unpredictability

  • Same query, different plans sometimes
  • Model updates break prompts
  • How much should I constrain the LLM?

Cost vs Quality

  • Better models = better plans = more $$$
  • Where's the sweet spot?
  • When to cache vs. when to re-plan?

What I Learned Building This

  1. Schema first. Define your domain before writing any LLM prompts. It saves debugging time later.
  2. Let LLMs reason, let DBs retrieve. Don't ask the LLM to search - ask it to plan. Databases are faster and cheaper.
  3. Cache aggressively, but carefully. Semantic caching sounds great until "free" and "paid" return the same results.
  4. Plan for failure. Qdrant down? Background worker. LLM hallucinates? Schema validation. Always have fallbacks.
  5. Start simple, add complexity when needed. I started with 1 collection. Now I have 4. The schema told me when to split.

Build something real. Break it. Fix it. That's how you learn.
