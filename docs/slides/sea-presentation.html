Schema: The Foundation

Schema ‚Üí Collections ‚Üí Query Planning ‚Üí Agentic Search

// schema.config.ts { "collections": { "tools": { "fields": ["name", "description"], "purpose": "semantic identity" }, "functionality": { "fields": ["features", "capabilities"], "purpose": "what it does" }, "useCases": { "fields": ["scenarios", "workflows"], "purpose": "when to use it" }, "interface": { "fields": ["deployment", "integration"], "purpose": "how to use it" } }, "vocabularies": { "categories": [...], "pricing": ["free", "freemium", "paid"] } }

Why Schema-Driven?

  • Defines collections: Schema decides what deserves separate embeddings
  • Enables routing: Query planner knows where to search
  • Validates LLMs: Controlled vocabularies catch hallucinations
  • Portable: New domain = new schema file

Why RRF?

Scale-Invariant

Works across collections with different score ranges. No normalization needed.

No Tuning Required

k=60 works well without domain-specific optimization. Just works out of the box.

Robust

Handles missing documents gracefully. A document only needs to appear in one source.

RRF lets us combine vector similarity scores with structured filters seamlessly

Open Research Questions

  1. Evaluation: How do we evaluate agentic search quality? No standard benchmarks.
  2. Learning: Can we learn query planning from user behavior? RLHF for planners?
  3. Architecture: How many agents is optimal? 3 nodes or 5? Or 10?
  4. Portability: Does schema-driven design work for other domains? Medical? Legal?
  5. Evolution: How to handle evolving schemas? New categories emerge over time.

These are genuine open questions. Let's collaborate!

Key Takeaways

For Researchers

  • Agentic search is interpretable and modular
  • Hybrid approaches often beat pure methods
  • We need better evaluation frameworks

For Practitioners

  • Schema-driven design enables portability
  • LLMs for reasoning, DBs for retrieval
  • Caching is essential for production
